# Kubernetes Sidecar Pattern – Nginx Log Shipping

## Objective

Demonstrate the Sidecar container pattern in Kubernetes by deploying
an Nginx web server container alongside a log-shipping container.

The sidecar container continuously reads access and error logs
generated by Nginx and outputs them every 30 seconds.

---

## Architecture

Single Pod: webserver

Containers:
- nginx-container (nginx:latest)
- sidecar-container (ubuntu:latest)

Shared Volume:
- emptyDir volume named shared-logs
- Mounted at /var/log/nginx in both containers

---

## Why Sidecar Pattern?

The sidecar pattern follows separation of concerns:

- Nginx handles web serving only
- Sidecar handles log shipping only

This improves modularity and maintainability.

---

## Volume Type Used

emptyDir

- Created when Pod starts
- Deleted when Pod terminates
- Shared between containers in same Pod
- Suitable for temporary data like logs

---

## Deployment

Apply the pod:

kubectl apply -f webserver.yaml

Verify:

kubectl get pods

Check logs:

kubectl logs webserver -c sidecar-container

---

## Key Concepts Demonstrated

- Multi-container Pod
- Volume sharing using emptyDir
- Sidecar container design pattern
- Container communication via shared filesystem
